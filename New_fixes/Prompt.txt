ISSUE
Clause parsing is now mostly correct, but repeated subclauses are still being merged incorrectly when roman numeral markers appear inline instead of at a clean line boundary.

Example problem shape:
- i) Group A ...
- ii) Group B ...
- iii) Group F ...

In some extracted blocks, only the first subclause is parsed correctly, while later sibling subclauses like ii), iii), iv) remain inside the previous clause/subclause content.

This causes wrong restructuring such as:
- clause.content contains "ii) Group B ... iii) Group F ..."
instead of:
- subclause (i)
- subclause (ii)
- subclause (iii)

IMPORTANT CONSTRAINT
Do NOT redesign the schema.
Do NOT change the existing hierarchy model.
Do NOT break currently correct parsing for:
- sentence markers
- clause markers
- already-correct subclause markers
- references like Clause (a), Subclause (i)

This must be a minimal, targeted fix for inline roman subclause boundaries only.

ROOT CAUSE
The parser still depends too much on line-start detection inside _process_article_text().
Subclause markers like i), ii), iii), iv) are sometimes arriving inline in the same text run, so later sibling subclauses are not split into separate parse units before parsing.

CURRENT RELEVANT LOGIC
- listgroup_to_lines() already inserts newlines before <li>
- _process_article_text() parses line by line
- clause/subclause detection depends on RE_LEGAL_MARKER and roman numeral handling
- current logic correctly identifies subclauses only when they appear as isolated parseable markers
