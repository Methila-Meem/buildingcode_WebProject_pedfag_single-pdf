ISSUE 1 — WRONG target_id PREFIX FOR SUBSECTIONS
==================================================

Current bad example:
{
  "text": "Subsection 3.7.2.",
  "kind": "Subsection",
  "number": "3.7.2",
  "target_id": "SEC-3-7-2",
  "resolved": false
}

Expected:
{
  "text": "Subsection 3.7.2.",
  "kind": "Subsection",
  "number": "3.7.2",
  "target_id": "SUBSEC-3-7-2",
  "resolved": true
}

RULE:
- Any reference with "kind": "Subsection" must resolve to a SUBSEC-* id, never SEC-*.
- Claude must check all extracted references globally and fix every subsection mismatch.
- If the referenced subsection exists in the document hierarchy, set:
  - target_id = SUBSEC-x-x-x
  - resolved = true
- If it truly does not exist, keep:
  - target_id = ""
  - resolved = false
- Never assign SEC-* to a Subsection reference.

This must apply everywhere:
- table references
- clause references
- sentence references
- article references
- note references
- any parsed reference object

      "kind": "Sentence",
      "raw": "Sentence 4.1.5.3.(1)",
      "number": "4.1.5.3.(1)",
      "target_id": "SENT-4-1-5-3-1",
      "resolved": true
    }
  }
}

Expected:
{
  "id": "TBL-208",
  "caption": {
    "raw": "Table 4.1.5.3. Specified Uniformly Distributed Live Loads on an Area of Floor or Roof Forming Part of Sentence 4.1.5.3.(1)",
    "table_number": "4.1.5.3.",
    "table_label": "Table 4.1.5.3.",
    "title": "Specified Uniformly Distributed Live Loads on an Area of Floor or Roof",
    "forming_part_of": {
      "kind": "Sentence",
      "raw": "Sentence 4.1.5.3.(1)",
      "number": "4.1.5.3.(1)",
      "target_id": "SENT-4-1-5-3-1",
      "resolved": true
    }
  }
}

RULES:
- table_number must preserve the trailing full stop when present in the source:
  - "4.1.5.3."
- table_label must also preserve it:
  - "Table 4.1.5.3."
- title must NOT start with a leading "."
- caption parsing must correctly split:
  1. table label
  2. title
  3. forming_part_of reference
- Apply this fix for all tables in the full document.

ISSUE 3 — FIGURE LINKING SCHEMA IS TOO WEAK
==================================================

Current figure example:
{
  "id": "FIG-55",
  "caption": "Figure 4.1.6.5.-A",
  "alt_text": "...",
  "image_key": "...",
  "image_path": "...",
  "page": 530
}

Current reference example:
{
  "text": "Figure 4.1.6.5.-A",
  "kind": "Figure",
  "number": "4.1.6.5.-A",
  "target_id": "FIG-55",
  "resolved": true,
  "source": "caption"
}

Problem:
- The system is confusing figure text/number matching, so hyperlinking often fails.
- Figure id alone is not enough.
- Figure caption string is not normalized enough for reliable matching.

REQUIRED SOLUTION
Refactor figure schema so figures have explicit parsed identifier metadata, similar to tables.

Use a schema like this:

{
  "id": "FIG-55",
  "caption": {
    "raw": "Figure 4.1.6.5.-A",
    "figure_label": "Figure 4.1.6.5.-A",
    "figure_number": "4.1.6.5.-A",
    "title": ""
  },
  "alt_text": "...",
  "image_key": "...",
  "image_path": "...",
  "page": 530,
  "page_span": [530],
  "reference_key": "Figure 4.1.6.5.-A"
}

REFERENCE RULES FOR FIGURES
- Every figure must have:
  - caption.raw
  - caption.figure_label
  - caption.figure_number
  - reference_key
- reference_key should be the canonical matching key used for hyperlink resolution.
- Reference objects pointing to figures should match primarily against:
  1. figure.reference_key
  2. figure.caption.figure_label
  3. figure.caption.figure_number

Reference example:
{
  "text": "Figure 4.1.6.5.-A",
  "kind": "Figure",
  "number": "4.1.6.5.-A",
  "lookup_key": "Figure 4.1.6.5.-A",
  "target_id": "FIG-55",
  "resolved": true,
  "source": "caption"
}

FIGURE MATCHING RULES
- Normalize figure matching consistently.
- Preserve suffixes like "-A", "-B"
- Do not confuse:
  - Figure 4.1.6.5.-A
  - Table 4.1.6.5.-A
- Figure references must only resolve against figures.
- Table references must only resolve against tables.
- Matching must be exact after normalization.

RECOMMENDED NORMALIZATION
Create canonical keys for tables and figures:
- Table canonical key:
  "Table 4.1.5.3."
- Figure canonical key:
  "Figure 4.1.6.5.-A"

Use canonical keys during resolution rather than fuzzy caption matching.
